Networking Authority Model

The pattern every multiplayer game should follow for authoritative state management — from local prototype to production networking.

Why this architecture exists

In any multiplayer game, a player's machine can be tampered with. If the client were allowed to directly change shared game state, cheating becomes trivial: a player could give themselves infinite gold in an RPG, teleport through walls in a shooter, or set the score to whatever they like in a sports game. The solution is the same across every genre: only one machine — the server — is permitted to change authoritative state. Every other machine (the clients) can only ask for things to happen. The server validates, executes, and tells everyone what the new reality is.

What actually happens, step by step

Every state-changing action in a multiplayer game follows this exact sequence — whether it's spending AP in a card game, firing a weapon in a shooter, or placing a unit in an RTS.

  1. 1
    The player does something on their screen.

    They interact with the UI — clicking a button, pressing a key, dragging a piece. This is a pure local event. Nothing in shared state has changed yet. e.g. clicking "Play Card" in this project, pulling the trigger in a shooter, hitting "End Turn" in a strategy game.

  2. 2
    The client sends a request to the server. It does not act itself.

    The client calls a request_* function on the server machine — not locally. It is asking permission, not taking action. In a local prototype where server and client share the same process, this is simulated by a local signal or direct function call. In production it becomes a network message. e.g. request_spend_ap(player_id, 2) in this project; request_fire_weapon(player_id, target_pos) in a shooter; request_move_unit(unit_id, tile) in an RTS. 

    # @rpc("any_peer") — any connected client can call this on the server.
# In the prototype, this annotation is omitted; a local signal stands in.
@rpc("any_peer", "reliable")
func request_spend_ap(player_id: int, amount: int) -> void:
    ...
  3. 3
    The server validates the request.

    The server checks whether the action is legal given the current authoritative state. If it isn't, the request is rejected and the client is told why. Nothing has changed yet. e.g. "Does this player have enough AP?" in this project; "Is the player's ammo count above zero?" in a shooter; "Is this unit actually yours, and is the tile reachable?" in an RTS.

  4. 4
    The server mutates its own copy of authoritative state.

    An _apply_* function runs on the server. This is the only place shared game state ever changes — one function, on one machine, with no exceptions. The server's copy is updated first, before any client knows. e.g. decrement AP tracker in this project; decrement ammo, apply hit detection in a shooter; update unit position on the board in a strategy game. 

    # No annotation. This function is never called remotely —
# it runs on the server only, called locally by request_*.
func _apply_spend_ap(player_id: int, amount: int) -> void:
    ...
  5. 5
    The server decides what each player is allowed to see.

    Before sending anything, the server constructs a separate payload for each recipient, filtering out information they shouldn't have. This is not optional — it is the only thing standing between the server's full picture and a curious player inspecting their network traffic. e.g. in this project, Player 2 gets a card count, not Player 1's actual hand; in poker, a player sees their hole cards but not the opponent's; in an MMO, a player only receives data for enemies within render distance.

  6. 6
    The server pushes each player their filtered slice of the new state.

    An _rpc_sync_* function is called — once per recipient when the payload differs per person, or broadcast to everyone when it doesn't. Nothing syncs automatically. If the server doesn't push it, no client will ever know it changed. e.g. AP tracker value goes to all players (same for everyone); hand contents go to each player separately (different per person). 

    # @rpc("authority") — only the server can invoke this on clients.
# Clients cannot call this themselves, so they cannot fake a sync.
# "reliable" — guaranteed delivery; a dropped packet means a permanently stale client.
@rpc("authority", "reliable")
func _rpc_sync_ap(value: int) -> void:
    ...

# How the server calls it:
_rpc_sync_ap.rpc(value)             # → runs on ALL connected peers
_rpc_sync_ap.rpc_id(peer_id, value) # → runs on ONE specific peer only
  7. 7
    Each client writes the incoming data into their local copy of state.

    When the sync message arrives, the client deserializes it and overwrites the relevant fields in their local state object. Their copy was stale a moment ago — now it reflects the server's authoritative version, for the fields they're allowed to see.

  8. 8
    The UI reacts to the updated local state.

    After writing the new state, the sync function fires an event (a Signal, a callback, an observer notification — whatever the framework uses). The UI hears it and redraws. The UI is always downstream of state. It reads state; it never writes it. e.g. the AP tracker widget redraws in this project; a health bar fills or drains in an RPG; the minimap updates to show a moved unit in an RTS.

Every multiplayer action follows this same eight-step path. The only thing that varies between actions is which fields get mutated in step 4 and what gets filtered per recipient in step 5.

Server State vs. Client State

Server (Authoritative) State Anything that, if tampered with, would give a player an unfair advantage or break the game for someone else. This state lives in a single authoritative object on the server. Every client has a local copy, but the server's version is the only truth.

In this project: AP tracker, each player's hand, deck, currency, turn order. In a battle royale: player positions, health, inventory. In an online board game: piece positions, whose turn it is, the score.
Client (Local) State Anything that only affects a single player's local experience and could not be exploited even if modified. This never touches the authoritative state object. It lives in UI nodes, local variables, or a separate local-only data structure.

Which card the player is currently reading in this project. Which enemy a player is hovering over in a shooter. Whether the settings panel is open in any game. Animation progress, screen shake timers, camera offsets.
A useful test: "If this value were randomly changed or deleted, would the other player notice or be harmed?" If yes, it's server state. If it only affects the local player's screen, it's client state.

The Multiple Copies Problem

The authoritative state object exists on every machine simultaneously — the server and each connected client all have their own copy in memory. They do not share a variable. The moment state changes, the copies diverge and must be explicitly re-synced by the server. In a local prototype, this is simulated by multiple peer contexts in the same process, but the rule holds: treat them as separate machines from day one, or the networking step will require a full rewrite.

The Three-Function Seam

Every state-mutating action passes through exactly three named layers. The naming convention makes each function's role unambiguous at a glance, and — crucially — makes the networking migration a mechanical step rather than a redesign.

Full call chain
request_*()
→
_apply_*()
→
_rpc_sync_*()
Prefix Who calls it What it does Future RPC annotation
request_* Any peer — client or server The public entrypoint. Validates the action against current state. Rejects if illegal. Calls _apply_* if valid. e.g. checking AP balance in a card game, checking ammo in a shooter, checking resource cost in an RTS. @rpc("any_peer")
_apply_* Server only — called locally Mutates authoritative state. The only place state changes. Calls _rpc_sync_* once mutation is done, for any fields clients need to know about. No annotation — server-local only, never called remotely.
_rpc_sync_* Server — sends to one or all clients Receives a serialized, filtered slice of state. Writes it into the local copy. Fires an event so the UI can react. @rpc("authority", "reliable")

RPC Annotations — What Each Part Means

In Godot 4, @rpc takes up to three arguments: @rpc(who_can_call, call_local?, transfer_mode). The same concepts exist in Unity (Mirror/Netcode), Unreal (RPCs), and most other multiplayer frameworks — only the syntax differs.

ArgumentOptionsMeaning
Who can call "any_peer" Any connected peer can invoke this function on a remote machine. Used on request_* — a client sends it toward the server.
"authority" Only the authority (server) can invoke this remotely. Used on _rpc_sync_* — prevents a client from faking a state sync to itself or others.
Call local "call_local" When calling .rpc(), also run the function on the calling machine. Useful when the server needs to update its own UI from the same sync function that clients use.
(omitted) Default behaviour: the function only runs on remote machines, not the caller. Use this when the server updates its own state directly inside _apply_* before calling the sync — which is the standard pattern here.
Transfer mode "reliable" Guaranteed delivery and ordering (equivalent to TCP). Use this for all state changes — a single dropped packet would leave a client permanently out of sync. e.g. a card being played, a unit moving, a player dying.
"unreliable" Fire-and-forget (equivalent to UDP). Acceptable only when the data will be resent so frequently that a missed packet doesn't matter. e.g. a player's real-time position in a shooter — it's updated every frame, so a dropped packet is invisible.

Calling conventions

MethodWhere it runsWhen to use
fn.rpc(...) All connected peers Broadcasting a change that is identical for everyone. e.g. a public score update, a shared timer, the AP tracker in this project.
fn.rpc_id(peer_id, ...) One specific peer Sending a filtered payload unique to one recipient. e.g. a player's private hand in a card game, a player's fog-of-war view in an RTS, a player's private inventory in an RPG.

Prototype Flow vs. Networked Flow

In a local prototype, server and clients share the same process. A local signal or direct function call is enough to simulate the boundary between them. When real networking arrives, that boundary becomes a physical network — and the callsite changes, but nothing else does. This is the entire point of the seam.

Now — local prototype

The local signal works as a stand-in because every peer is in the same process. The _on_* callback is the effective request boundary — it just arrives via signal instead of over a network.

The migration cost is intentionally low. Add @rpc("any_peer") to each request_* function. Change each callsite from a signal emit to .rpc_id(server_id, ...). Remove the local signals that were standing in. The _apply_* and _rpc_sync_* layers are entirely untouched.

Full Code Example — Spending AP (this project)

Tracing one action through all three layers with annotations shown. The same structure applies in any game: swap "spend_ap" for "fire_weapon" in a shooter, "place_building" in a city builder, or "fold_hand" in an online poker game — the skeleton is identical.

Prototype (today — GDScript, Godot 4)

## ----- On the client (same process — no real network yet) -----

# Player clicks "Play Card". Cost is 2 AP.
# A local signal stands in for the future RPC call.
SignalBus.spend_ap_requested.emit(my_peer_id, 2)


## ----- In GameManager (server context) -----

# Signal callback — this is the request arriving, simulated locally.
func _on_spend_ap_requested(player_id: int, amount: int) -> void:
    if not can_spend_ap(player_id, amount):
        SignalBus.ap_spend_failed.emit(player_id)   # reject
        return
    _apply_spend_ap(player_id, amount)


# Authority-only mutator. No annotation needed yet — server-local only.
func _apply_spend_ap(player_id: int, amount: int) -> void:
    if _is_player_1(player_id):
        _game_state.ap = clamp(
            _game_state.ap - amount,
            -max_ap_value,
            max_ap_value
        )
    else:
        _game_state.ap = clamp(
            _game_state.ap + amount,
            -max_ap_value,
            max_ap_value
        )
    # AP tracker is public info — broadcast the same value to everyone.
    _rpc_sync_ap.rpc(_game_state.ap)


# "authority" means only the server can call this on clients.
# Clients cannot fake a sync to themselves.
@rpc("authority", "reliable")
func _rpc_sync_ap(value: int) -> void:
    _game_state.ap = value
    SignalBus.ap_moved.emit(value)  # UI redraws here

## ----- On the CLIENT machine -----

# Same button press. But now we send a message to the server machine,
# instead of emitting a local signal.
game_manager.request_spend_ap.rpc_id(SERVER_PEER_ID, my_peer_id, 2)

    # ... same mutation logic ...
    _rpc_sync_ap.rpc(_game_state.ap)

_apply_spend_ap and _rpc_sync_ap are byte-for-byte identical between the prototype and the networked version. The only changes are one annotation added to request_* and one callsite swap from signal to .rpc_id(). Everything else was already in the right shape.

Information Hiding — Filtered Sync Payloads

The sync step is the only place where hidden information could be accidentally leaked. The server must construct a different outgoing payload per recipient before any _rpc_sync_* call that involves private data. This is not about trust — it is about what is even sent over the wire. A sufficiently motivated player will inspect their own network traffic. The server must never put something in the packet that the recipient isn't supposed to see.

In poker: the server sends each player their hole cards but nobody else's. In an FPS with fog of war: the server only sends enemy positions if the enemy is visible to that player. In a hidden-role game like Werewolf: only the werewolves receive a message telling them who the other werewolves are. 

# Server iterates every peer and builds a tailored payload for each.
func _sync_player_state_to_all_peers() -> void:
    for peer_id in NetworkManager.connected_players:
        var opponent_id: int = _opponent_id(peer_id)

        _rpc_sync_player_state.rpc_id(
            peer_id,
            _serialize_own_state(_game_state.player_states[peer_id]),  # full view
            _game_state.player_states[opponent_id].get_public_view()   # filtered
        )


# Full state — only ever sent to the player who owns it.
func _serialize_own_state(state: PlayerState) -> Dictionary:
    return {
        "currency": state.currency,
        "hand":     state.hand,     # actual card data
        "deck":     state.deck,
        "discard":  state.discard,
    }


# Public view — sent to the opponent.
# Cards are stripped. Only counts are exposed.
func get_public_view() -> Dictionary:
    return {
        "currency":   currency,
        "hand_count": hand.size(),   # how many, not which ones
        "deck_count": deck.size(),
    }

The Rules at a Glance

  • Only _apply_* functions write to authoritative state. No exceptions, anywhere in the codebase.
  • request_* is the sole public entrypoint for any state change. It validates, then delegates. It never mutates directly.
  • After mutating state, push it to clients via _rpc_sync_*. Clients will never see a change unless the server explicitly sends it.
  • Build a separate payload per recipient whenever the data differs per player. Never trust the client to ignore data it wasn't supposed to receive.
  • Client state — UI, animation, hover effects — lives in UI nodes or local variables. It never touches the authoritative state object.
  • The UI reacts to sync events. It never reads state on a timer and never writes state at all.
  • In a local prototype, local signals or direct calls stand in for request_* RPC calls. Treat the boundary as if the network exists from day one.
