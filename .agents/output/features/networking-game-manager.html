FEATURE 2026-05-22

GameManager Networking Finalization

The lobby is implemented and loads into the game scene. GameManager._ready() has the skeleton for player initialization but has two foundational gaps that must be resolved before any of the remaining @rpc work is wired up. Fix these two spots first — everything else depends on them being correct.

1 Seat / peer_id mismatch in _initialize_player
The bug — game_manager.gd:40–44

player_states is populated with peer_id as the key (e.g. 1 for the server, a large random number for the client). But every game logic function — _apply_add_currency, _apply_switch_turn, _print_players_hands — accesses player_states[_game_state.active_player], where active_player is PlayerSeat.PLAYER_ONE = 0 or PlayerSeat.PLAYER_TWO = 1. Those keys will never match the actual peer IDs — the lookup will silently return null or crash at runtime.

The fix

player_states should stay keyed by PlayerSeat.Type — the game logic is already written that way. _initialize_player needs to assign a seat rather than use the raw peer_id as the key. GameState also needs a reverse lookup so you can translate an incoming peer_id into a seat later (e.g. when an RPC arrives from a specific peer).

Changes needed:

game_state.gd — add a seat lookup alongside player states:

var player_states: Dictionary[int, PlayerState] = {}  # PlayerSeat.Type → PlayerState
var peer_to_seat: Dictionary[int, int] = {}            # peer_id → PlayerSeat.Type

game_manager.gd — assign seats in order of connection:

func _initialize_player(peer_id: int) -> void:
    var seat: PlayerSeat.Type = PlayerSeat.PLAYER_ONE \
        if _game_state.player_states.is_empty() \
        else PlayerSeat.PLAYER_TWO
    _game_state.peer_to_seat[peer_id] = seat
    _game_state.player_states[seat] = PlayerState.new()
    _game_state.player_states[seat].deck = test_deck.cards.duplicate()
2 Clients never get _initialize_player called
The gap — game_manager.gd:28–36

_initialize_player only runs on the server. Clients load the game scene, _subscribe_to_game_signals connects them to all the same signals, but player_states is empty on their side. Any _apply_* function that touches player_states will fail on the client.

The fix

After the server initializes a player, broadcast it to all clients with an @rpc. The client just mirrors the seat assignment — no logic, just state setup. Note that clients intentionally receive no deck here; card state lives on the server and syncs separately. They do need the PlayerState entry to exist so UI reads don't fail.

Changes needed in game_manager.gd:

@rpc("authority", "call_local", "reliable")
func _rpc_sync_player_init(peer_id: int, seat: PlayerSeat.Type) -> void:
    if multiplayer.is_server():
        return  # server already handled this in _initialize_player
    _game_state.peer_to_seat[peer_id] = seat
    _game_state.player_states[seat] = PlayerState.new()
