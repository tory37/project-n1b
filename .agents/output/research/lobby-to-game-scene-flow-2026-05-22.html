Section II — Scene Architecture Decision: Wrapper or Flat?

You asked whether there should be a lobby scene separate from the lobby UI. Two patterns exist:

Pattern A: Flat — lobby_ui.tscn IS the scene (Recommended)

The root node of lobby_ui.tscn is a Control set to Full Rect. This is a valid Godot scene root. You set it as the main scene and you're done. No wrapper needed.

Why it works: The lobby has no 3D world, no camera, no environment — it's pure UI. A Control as the scene root is idiomatic Godot for full-screen UI scenes.

Tradeoff: If you ever want background artwork (an animated 3D table, a skybox, etc.) behind the lobby menu, you'd need to restructure then. That's a future concern — do it flat now.

Pattern B: Wrapper — lobby.tscn (Node3D) contains lobby_ui.tscn as child

A Node3D root called Lobby with a Camera3D, background environment, and lobby_ui.tscn instanced as a child.

Why you'd want this: If you want the same 3D table visible behind the lobby UI (visual consistency with the game). Adds depth and polish.

Tradeoff: More setup now for a feature you haven't designed yet. Not worth it at this stage.

Decision: Go with Pattern A. lobby_ui.tscn becomes the main scene. If you add 3D background later, you convert it then — it's a 10-minute change.

Section III — The RPC Transition Pattern (How It Actually Works)

This is the most important thing to understand about what's already in the code:

# lobby_ui.gd:85-93

func _on_start() -> void:
    if not NetworkManager.is_server:
        return
    _start_game.rpc()           # ← calls _start_game() on ALL connected peers

@rpc("authority", "call_local", "reliable")
func _start_game() -> void:
    get_tree().change_scene_to_file("res://src/levels/game/game.tscn")

The @rpc decorator flags break down as:

FlagMeaning
"authority"Only the server (authority) may call this across the network.
"call_local"The server also executes it locally — not just on remote peers.
"reliable"Guaranteed delivery (ENet reliable channel). Critical for scene transitions.

The net effect: when the host presses Start Game, _start_game() runs on every machine (host included). All peers simultaneously call change_scene_to_file. The MultiplayerSpawner in game.tscn then handles spawning player objects for each peer ID that was in NetworkManager.connected_players.

Caveat — RPC owner: _start_game.rpc() works because lobby_ui.gd is a node in the scene tree at call time. After change_scene_to_file, that node is freed. This is fine — the RPC fires before the scene swap completes.

Section IV — Bugs in game_manager.gd That Will Block You

The lobby → game transition will work, but game_manager.gd has several syntax/logic errors that will crash on load. These must be fixed before testing the flow end-to-end.

Bug 1: Unclosed constructor call (syntax error) CRASH

game_manager.gd:15

The DeckData.new( is an unclosed call — Godot will refuse to parse this file entirely. Since test_deck is exported and assigned in the scene, the initializer isn't needed anyway.
