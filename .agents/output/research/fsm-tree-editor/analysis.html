Replacing the GraphEdit FSM editor with a tree visualization

The question

The current FSM authoring view (src/editor/fsm/fsm_editor.gd) is built on Godot's GraphEdit / GraphNode. Because GraphEdit is a free-floating node canvas, it can't lay the machine out as a tree — even though, in practice, these machines will be trees that occasionally loop back on themselves. The ask: what would it take to build a custom editor that visualizes the FSM structure as a tree?

The good news: the data model is already editor-agnostic

Nothing about the FSM is tied to GraphEdit. The renderer is fully decoupled from the model, so a tree view is a parallel renderer, not a rewrite.

  • Runtime (finite_state_machine.gd) only reads entry_state and follows each state's state_change_requested signal. It never sees the editor.
  • The graph shape comes entirely from introspection. FiniteState exposes get_successor_properties() (typed FiniteState exports = edges) and get_data_properties() (everything else = inline fields). Any view can walk that.
  • The plugin shell is reusable as-is. The inspector button (fsm_inspector_plugin.gd) emits edit_graph_requested(fsm, host); the plugin (fsm_editor_plugin.gd) calls open_fsm(fsm, host) on whatever Control it hosts. Swap the Control, keep the wiring.
  • The only model touchpoint to drop: the graph view writes a graph_position meta onto each state for free-canvas layout. A tree view auto-lays-out, so it just ignores (or stops writing) that meta.

So roughly everything except the rendering/interaction layer is salvageable: host resolution, save / dirty / auto-save, the state-script picker, deletion semantics, the inspector entry point.

The one real tension: it's a graph, not a tree

The existing code goes out of its way to handle two things that a literal tree cannot represent natively:

  • Diamonds — two branches converge on the same state. The graph view de-dups by instance identity (_node_by_state) so one node draws with multiple inbound edges.
  • Cycles — a successor points back "up" the graph (the loop-back you described). clone_graph() and _print_state() both guard against recursing forever.

A tree shows each node once per parent path. So a shared or looping target has only two honest renderings, and which one you pick drives the entire effort estimate:

RenderingWhat the user seesCost
Reference marker Canonical node shown once (at its shallowest occurrence). Every other inbound edge becomes a clickable leaf: ↺ → Combat. Click to jump to the real node. Cheap, robust, scales to deep graphs, collapsible.
Drawn back-edge Node laid out top-down by depth; forward edges form the tree skeleton; loop/cross edges drawn as curved arrows looping back. You literally see where it loops. Expensive — you own layout, edge routing, hit-testing.

This is the pivotal decision. "Show me a clean collapsible hierarchy" and "show me the loop physically curving back" are two different products with a roughly 5× effort gap.

Three approaches

A Godot Tree control (collapsible reference tree)

Render into a built-in Tree. Each state is a TreeItem; each successor export is a child item labeled by its branch name (on_empty →); revisited/looping targets become reference leaves. The Tree gives you collapsing, scrolling, and auto-layout for free.

  • Walk entry_state → successors exactly like _rebuild_node does today, emitting TreeItems instead of GraphNodes; keep a TreeItem ↔ state map.
  • Inline data: either Tree cell editors (check / range / text modes) or — cleaner — a detail panel beside the tree showing the selected state's data exports as real Controls (reuse _build_value_editor almost verbatim).
  • Per-item buttons via TreeItem.add_button: "+ add successor", "delete".
  • Rewire to an existing state: a picker listing existing states (replaces GraphEdit's drag-to-connect).
Effort: low–moderate. A few hundred lines, mostly mechanical. Loops/diamonds shown as reference leaves. The biggest judgment call is "Tree cell editors vs. side detail panel" (detail panel is nicer). Best effort-to-value if reference markers are acceptable.
B Custom-drawn tree (Control._draw) with auto-layout + back-edges

A bespoke canvas: run a layered tree-layout pass (assign depth, pack siblings — Reingold–Tilford / Sugiyama style), then draw node boxes, forward edges as the skeleton, and loop/cross edges as curved arrows. Add pan / zoom / hit-testing / selection popups.

  • Layout engine: depth assignment + sibling packing + back-edge detection (entry-rooted DFS).
  • Render pass: boxes, labels, forward + back edge routing, arrowheads.
  • Interaction: pick/select via hit-testing, popups or a detail panel for editing, buttons for add/delete.
  • Reuse the same data walk, picker, save/dirty as A.
Effort: high. This is building a mini graph-layout engine. The only approach that delivers the "tree that visibly loops back" visualization in your description.
C Keep GraphEdit, add an auto tree-layout command

Leave GraphEdit; add a "Tidy / Auto-layout" button that runs a proper layered top-down layout and writes position_offset, instead of today's naive per-depth column packing in _rebuild_node.

Effort: lowest. But it does not change the interaction model you're unhappy with — it's still a free canvas you drag around; it just arranges itself tree-like on demand. Loops still draw as GraphEdit's curved edges. Listed for completeness; probably not what you want.

Where it opens ("some window")

  • Bottom panel (today) — zero change to the shell.
  • Main-screen plugin — implement _has_main_screen(); it gets a top tab like 2D / 3D / Script. Most room, feels like a first-class editor.
  • Floating Window popped from the inspector button — closest to literally "opens in a window." Easy: wrap the view Control in a Window and popup().

This is independent of A/B/C — any renderer drops into any of these shells.

Recommendation

Approach A (Godot Tree control), in a floating Window or main-screen tab — if showing loops/diamonds as clickable reference markers is acceptable. It reuses nearly all existing logic, auto-lays-out, collapses deep branches, and is the natural fit for "these are trees." Reach for Approach B only if the loop edges drawing back is the actual point of the visualization.

The decision I need from you to size this concretely: when a branch loops back or two branches converge, do you need to see the edge drawn (→ B), or is a clickable ↺ → Combat reference marker enough (→ A)?
