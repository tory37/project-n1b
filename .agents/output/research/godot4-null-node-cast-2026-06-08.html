Why $Node as CustomClass Returns Null in Godot 4

I. The Problem

In game_manager.gd, the following pattern returns null at runtime despite the node existing in the scene tree and its script correctly extending GamePhaseState:

_phase_nodes = {
    GamePhase.START: $FSM/Start as GamePhaseState,  # null
    ...
}

The error: "can't call setup on Nil" — meaning the dictionary value is null by the time .setup() is called. This persisted after moving initialization from @onready into _ready() directly.

II. Root Causes Found

LIKELY CAUSE 1 The as keyword cast is a known Godot 4 bug

Godot issue #41777 documents that casting a node to a custom class_name type using the as keyword returns null — both in @onready declarations and in _ready() body assignments. The node IS found by $; it's the as cast step that silently fails.

Affects: both @onready var x: T = $Path as T and var x = $Path as T in _ready()

Key observation: The bug does NOT occur when you cast to the built-in Node type. It specifically fails with class_name-declared types. The fix is to remove the as keyword and let GDScript's type inference handle the assignment.

Evidence: Class cache (.godot/global_script_class_cache.cfg) was inspected — all entries are correct. GamePhaseState is registered at the new path, no stale entries from the deleted phase/ folder. The inheritance chain is valid. This rules out cache corruption and points squarely at the as cast.

BUG 2 Wrong script overrides in game.tscn — two nodes have phase scripts

Two nodes in game.tscn have accidental FSM script overrides, likely from an errant drag in the editor:

NodeWrong Script AssignedExpected
Table (Node3D instanced scene)main.gd (GamePhaseMain)No script override
GameOverlayUi (instanced scene)draw_cards.gd (GamePhaseDrawCard)No script override

This means there are two live instances of GamePhaseMain and GamePhaseDrawCard in the scene tree — one on the FSM nodes (correct) and one on unrelated nodes (wrong). These wrong instances have _game_manager = null (never had .setup() called) and will cause null-reference errors if FSM transitions ever reach them.

BUG 3 My previous edit introduced a broken node path

In the first fix attempt, I changed $FSM/ResolvingCard to $FSM/PlayingCard based on an incorrect grep result from an older version of the scene. The current game.tscn on disk has the node correctly named ResolvingCard. This means my edit broke the fourth entry in the dictionary.

III. Confidence Assessment

Cause 1 (as-cast bug): High Probability. Confirmed by Godot issue #41777, which documents this exact pattern. What is not yet confirmed is whether this is the ONLY cause — a targeted debug print (see Section V) would definitively separate "node not found" from "cast returning null."

Cause 2 (wrong script overrides): Confirmed as a bug by direct file inspection, though not confirmed as the cause of the null dictionary values. Must be fixed regardless.

Cause 3 (PlayingCard path): Confirmed by reading the current scene file. Definitely broken by my edit.

IV. The Fix

Three changes are required:

1 Remove as casts — use plain get_node assignment

Change Dictionary[GamePhase, GamePhaseState] to an untyped Dictionary and remove all as GamePhaseState casts. The runtime type is still correct — the nodes are GamePhaseState instances.

var _phase_nodes: Dictionary  # drop typed params to avoid secondary issues

func _ready() -> void:
    _game_fsm = FiniteStateMachine.new()
    _phase_nodes = {
        GamePhase.START:          $FSM/Start,
        GamePhase.DRAW_CARD:      $FSM/DrawCard,
        GamePhase.MAIN:           $FSM/Main,
        GamePhase.RESOLVING_CARD: $FSM/ResolvingCard,
    }
    _phase_nodes[GamePhase.START].setup(self)
    _phase_nodes[GamePhase.DRAW_CARD].setup(self)
    _phase_nodes[GamePhase.MAIN].setup(self)
    _phase_nodes[GamePhase.RESOLVING_CARD].setup(self)

If type safety on the dict is important, store the refs as plain Node references first and cast later when consuming them — not at assignment time.

2 Remove wrong script overrides from game.tscn

In the Godot editor, select the Table node in game.tscn and detach the script (Inspector → Script → right-click → Detach Script). Do the same for GameOverlayUi. These nodes should have no script override; their behaviour comes from their instanced scenes.

3 (Already reverted by fix 1) — ResolvingCard path

The rewrite in step 1 restores $FSM/ResolvingCard, overwriting my earlier incorrect change to $FSM/PlayingCard.

V. Debug Confirmation (Optional)

If the null persists after applying the fix, add these prints at the top of _ready() to split the two possible failure modes:

var start_raw = $FSM/Start
print("raw node: ", start_raw)                              # null = path wrong
print("is GamePhaseState: ", start_raw is GamePhaseState)  # false = cast would fail
print("script: ", start_raw.get_script() if start_raw else "N/A")
  • If start_raw is null → the path is wrong or scene didn't load; verify in editor's Remote Debugger scene tree.
  • If start_raw is not null but is GamePhaseState is false → engine-level type identity issue; try clearing .godot/ cache and reimporting.
  • If both look correct → issue is in typed Dictionary coercion; confirm the untyped approach works.
