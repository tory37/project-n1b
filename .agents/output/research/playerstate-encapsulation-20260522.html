PlayerState Encapsulation vs. Flat Fields in GameManager

The Question: Is it good practice to keep per-player data in a PlayerState object? Or should GameManager hold raw/flat fields and surgically pass only what each peer needs — avoiding the deck-hiding problem that comes with sending a full PlayerState?

Short Answer: PlayerState encapsulation is the correct pattern. The deck-hiding problem is not caused by using a PlayerState object — it is caused by sending the wrong thing over the wire. That is a sync-layer problem, solved at the RPC/serialization boundary, not at the storage layer. Flat fields in GameManager do not remove the problem; they just make it harder to fix cleanly.

I. The Core Misunderstanding — Storage vs. Transport

The intuition behind flat fields is: "if I only have one variable per thing, I can just choose not to send the sensitive one." That intuition is correct — but it applies equally to encapsulated objects. The storage shape does not determine what gets sent; the RPC call does.

Approach How deck hiding works Tradeoff
Flat fields in GameManager
p1_hand, p1_deck, p2_hand, p2_deck… 		Manually pass each field that is allowed. You write rpc("sync_p2_state", p2_hand_count, p2_currency) and omit p2_deck. Works, but every new field requires manually auditing every RPC call site. GameManager becomes a God Object. [State Management Patterns — peerdh.com] calls this "scattered state" — it is the #1 root cause of sync bugs in multiplayer games because there is no single place to enforce what is and is not public.
PlayerState object
player_states[seat]: PlayerState 		Add a method to PlayerState: get_public_view() -> Dictionary that strips sensitive fields before serialization. The RPC always calls rpc("sync_opponent_state", opponent.get_public_view()). The rule is in one place, enforced at the class boundary. Slightly more code up front. Pays off immediately on the first field you add to PlayerState — you do not have to audit every RPC call site.

II. Industry Precedent — All Major Engines Use Encapsulated PlayerState

This is not a marginal design choice. It is the dominant pattern across every major engine and authoritative networking framework.

  • Unreal Engine has a dedicated APlayerState class that is replicated to all peers. The engine separates it from AGameState (global state) by design. [Cedric Neukirchen — Multiplayer Compendium]
  • Real-time card games (.NET architecture guide) use a state-sync model where "the server continuously transmits authoritative state snapshots to clients." The snapshot shape per-player is a bounded, structured object — not a flat dump. [DevelopersVoice — Real-Time Multiplayer Card Games]
  • Server-authoritative fog of war (the exact deck-hiding problem this project faces): "your authoritative node needs to be a bit smarter and only sends the game state that a player should know. If you have done this, then even if the cheater lifts fog of war, the game state will not be revealed." [Ruoyu Sun — Game Networking Demystified] The encapsulated object is what makes the server "smarter" in a maintainable way.

III. The Working Stack — Three Evidence-Based Approaches

Method Theoretical Basis Practical Implementation (Godot 4)
PlayerState + get_public_view() Recommended Information hiding at the class boundary. The object owns the rule of what is public. Follows the Single Responsibility Principle — PlayerState is responsible for knowing its own data and what it is safe to broadcast. PlayerState keeps full private fields (deck, hand). Add: 
func get_public_view() -> Dictionary:
    return { "hand_count": hand.size(),
             "currency": currency }
Server RPC sends get_public_view() to opponents, full state to owner. [StraySpark — Authoritative Server Guide]
State Snapshot Sync Valid alternative Server broadcasts authoritative state snapshots each tick or on change. [Ruoyu Sun] The snapshot content is filtered per-recipient. Works well for discrete turn-based games. On each state change, server calls two RPCs: _rpc_sync_own_state.rpc_id(owner_id, full_state) and _rpc_sync_opponent_state.rpc_id(opponent_id, public_view). No polling required — only fire on mutation.
MultiplayerSynchronizer (limited use) Partial fit Godot 4's built-in property replication. Works for primitives. [Godot Forum] Does not support per-recipient filtering — it broadcasts to all peers equally. [GodotAwesome 2025] Cannot hide the deck using this alone. Use RPCs for anything that needs per-player visibility control.

IV. Engagement Theater vs. Active ROI

Applied to this design question: what looks like a simplification but creates hidden debt, versus what adds a small upfront cost and pays off immediately.

Engagement Theater — Flat fields in GameManager
  • Feels simpler: fewer files, fewer classes.
  • Every new player attribute (health, status effects, played cards) must be added to GameManager as a new pair of variables (p1_x, p2_x). The file grows into a God Object. [peerdh.com]
  • Information hiding logic is spread across every RPC call site. One forgotten omission leaks the deck. There is no single place to audit.
  • Adding a third player (or even a spectator) requires a surgery of the entire GameManager surface.
Active ROI — PlayerState with get_public_view()
  • Adding a new attribute (e.g., mana) touches one class. It is private by default; you opt in to making it public in one method.
  • RPC call sites never need to change — they always call get_public_view(). The rule lives in one place.
  • The GameManager stays lean: it orchestrates state transitions, it does not own per-player data.
  • Scaling to a third player means adding a third entry to player_states — no structural changes.

V. The 5 Inviolable Laws

  1. Information hiding is a sync-layer problem, not a storage-layer problem. Whether you use flat fields or encapsulated objects, you still decide what to put in each RPC. The object just enforces that decision consistently. [Ruoyu Sun]
  2. The server is the only node that holds full PlayerState. Clients receive either their own full state or their opponent's public view — never the opponent's full state. [DevelopersVoice]
  3. Do not use MultiplayerSynchronizer for hidden state. It broadcasts to all peers without per-recipient filtering. Use manual @rpc calls for anything that needs visibility control. [Godot Forum]
  4. One class owns the definition of "what is public." Put get_public_view() on PlayerState. Never let GameManager decide field-by-field at each call site — that path leads to leaks.
  5. GameManager orchestrates; PlayerState owns. GameManager's job is to sequence game logic (turn flow, AP tracker, phase FSM). PlayerState's job is to hold and expose player data. Keep them separated or GameManager becomes the bottleneck for every feature. [peerdh.com]

VI. Applied to This Project — What Changes

PlayerState is correct as-is for storage. The one addition needed is a method that defines what is safe to send to opponents: 

# player_state.gd — add this method
func get_public_view() -> Dictionary:
    return {
        "hand_count": hand.size(),
        "discard_count": discard.size(),
        "currency": currency,
    }
    # deck is intentionally excluded

Then in GameManager, the RPC that syncs state to each client sends the appropriate view: 

# On the server, after any state mutation:
for peer_id in NetworkManager.connected_players:
    var seat := _game_state.peer_to_seat[peer_id]
    var own_state := _game_state.player_states[seat]
    var opponent_seat := (seat + 1) % 2
    var opponent_public := _game_state.player_states[opponent_seat].get_public_view()
    _rpc_sync_state.rpc_id(peer_id, own_state_as_dict, opponent_public)

Nothing about PlayerState needs to change. The information hiding happens at the boundary where data leaves the server, not in how data is stored.
