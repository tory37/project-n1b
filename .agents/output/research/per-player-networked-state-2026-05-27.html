Per-Player Networked State in Godot 4

How to adapt the NetwrkedState template for state where each player has their own value — and how to sync it correctly for both public and private information.

I. The Core Distinction: Scalar vs. Per-Player State

Your current template is built around scalar state — one authoritative value shared identically by all clients (_value: int). That works perfectly for TurnNumber, ActivePlayer, and TurnOrder.

Per-player state is different: the server holds one value per connected peer, and depending on whether that value is public or private, it gets synced differently.

State Type Server Storage Sync Target Examples in Your Game
Scalar (global shared) _value: T rpc() → all peers TurnNumber, ActivePlayer, TurnOrder
Per-player, public _values: Dictionary[int, T] rpc(peer_id, val) → all peers Action Points, Currency/Score
Per-player, private _values: Dictionary[int, T] rpc_id(peer_id, val) → owner only + optional count broadcast Hand (card contents), Deck contents

II. Is Dictionary[peer_id → value] the Standard Pattern?

Short answer: Yes. It is the canonical server-side pattern in Godot 4.

The official Godot docs demonstrate exactly this: var players = {} keyed by peer_id. [Godot Docs — High Level Multiplayer]

The StraySpark authoritative server guide uses the same pattern: var player_states: Dictionary = {} # peer_id → PlayerState and retrieves per-request with multiplayer.get_remote_sender_id(). [StraySpark — Authoritative Server]

Your own game_state.gd already does this at the non-networked layer: var player_states: Dictionary[int, PlayerState] = {}. The migration is consistent with your existing mental model.

The Alternative: Per-Player Nodes (MultiplayerSpawner)

Godot's scene replication system (MultiplayerSpawner + MultiplayerSynchronizer) lets you spawn one node per player and let Godot handle sync automatically via property replication. [Godot Blog — Scene Replication]

This is best suited for real-time games with many players where each player entity has transform, physics, and animation state. For your case — a 2-player turn-based card game using custom RPC state nodes — the dictionary approach is the right call for these reasons:

  • Your architecture is already built around explicit request/apply/sync RPCs, not Godot's auto-sync. Mixing them would add confusion.
  • Spawned nodes require MultiplayerSpawner configuration and authority assignment at runtime — more machinery for no gain.
  • A dictionary centralizes all player state in one place, which is easier to serialize, inspect, and debug.
  • Community consensus for turn-based card games points to plain RPC patterns over scene replication. [Godot Forum — Turn-Based Strategies]

III. The Two Per-Player Template Variants

Your template needs two new variants. The structure is identical to the existing template — the only things that change are storage shape and sync target.

Variant A — Public Per-Player State (e.g. Action Points)

Everyone can see every player's AP. The server holds a dict; sync broadcasts the updated peer_id + value to all clients.

class_name ActionPointsNetworkedState
extends Node

# Dictionary[peer_id → ap_value]. Server-side source of truth.
var _values: Dictionary[int, int] = {}

func get_value(peer_id: int) -> int:
    return _values.get(peer_id, 0)

# --- Public Methods ---

func set_value(peer_id: int, new_value: int) -> void:
    if not multiplayer.is_server():
        push_error("Only the server can set AP directly")
        return
    _apply_set(peer_id, new_value)

@rpc("any_peer", "reliable")
func request_spend(amount: int) -> void:
    if not multiplayer.is_server():
        return
    var sender_id: int = multiplayer.get_remote_sender_id()
    var current: int = _values.get(sender_id, 0)
    var new_value: int = current - amount
    if not _validate_spend(sender_id, amount):
        _sync_spend_failed.rpc_id(sender_id)
        return
    _apply_set(sender_id, new_value)

# --- Private Methods ---

func _validate_spend(peer_id: int, amount: int) -> bool:
    return amount <= _values.get(peer_id, 0)

func _apply_set(peer_id: int, new_value: int) -> void:
    if not multiplayer.is_server():
        return
    # Broadcast to ALL peers — AP is public information
    _sync_value.rpc(peer_id, new_value)

# All peers (including server) update their local copy
@rpc("authority", "reliable")
func _sync_value(peer_id: int, new_value: int) -> void:
    _values[peer_id] = new_value
    SignalBus.action_points_synced.emit(peer_id, new_value)

@rpc("authority", "reliable")
func _sync_spend_failed() -> void:
    SignalBus.action_points_spend_failed.emit()
Key difference from your scalar template: _sync_value now takes (peer_id: int, new_value: int) as arguments. Every client — including the server — stores the full _values dictionary so local UI can query any player's AP.

Variant B — Private Per-Player State (e.g. Hand)

The hand contents are secret. The server holds the full dict; sync sends contents only to the owning peer. Other peers receive only the count (so they can render "3 cards face-down").

class_name HandNetworkedState
extends Node

# Server holds all hands. Clients hold only their own.
var _values: Dictionary[int, Array] = {}  # peer_id → Array[card_resource_path]

func get_hand(peer_id: int) -> Array:
    return _values.get(peer_id, [])

# --- Public Methods ---

func set_hand(peer_id: int, cards: Array) -> void:
    if not multiplayer.is_server():
        push_error("Only the server can set a hand directly")
        return
    _apply_set(peer_id, cards)

@rpc("any_peer", "reliable")
func request_draw(count: int) -> void:
    if not multiplayer.is_server():
        return
    var sender_id: int = multiplayer.get_remote_sender_id()
    # ... draw logic, add to _values[sender_id] ...
    _apply_set(sender_id, _values[sender_id])

# --- Private Methods ---

func _apply_set(peer_id: int, new_cards: Array) -> void:
    if not multiplayer.is_server():
        return
    _values[peer_id] = new_cards

    # Private: send full hand ONLY to the owning peer
    _sync_hand.rpc_id(peer_id, new_cards)

    # Public: tell all other peers the count only
    _sync_hand_count.rpc(peer_id, new_cards.size())

# Received only by the owning peer
@rpc("authority", "reliable")
func _sync_hand(cards: Array) -> void:
    # Client stores only their own hand — no peer_id arg needed
    SignalBus.hand_synced.emit(cards)

# Received by all peers (for face-down rendering)
@rpc("authority", "reliable")
func _sync_hand_count(peer_id: int, count: int) -> void:
    SignalBus.hand_count_synced.emit(peer_id, count)
Why no peer_id in _sync_hand? Because it's sent with rpc_id(peer_id, ...), the recipient is always the owning player. They don't need to be told whose hand it is — it's theirs.

IV. The Hidden State Problem — A Worked Example

The most important invariant for card games: the server must never broadcast private card state to all clients. A cheating client can read any RPC that arrives at their machine. This is documented explicitly in the Godot forum's card game thread. [Godot Forum — 3D Card Game Sync]

Operation Server Action Sync to Owner Sync to Others
Draw card Add card to _values[peer_id] rpc_id(peer_id) with full updated hand rpc() with new hand count
Play card Remove card from _values[peer_id] rpc_id(peer_id) with updated hand rpc() with card identity (it's now revealed) + new count
Discard hidden Remove card from _values[peer_id] rpc_id(peer_id) confirming discard rpc() with new hand count only
Initial hand deal Populate _values for all peers rpc_id(peer_id) per player with their specific hand rpc() with each player's starting count
Never call: _sync_hand.rpc(peer_id, cards) — this sends every player's hand to every client. Always use rpc_id for private contents.

V. Mapping Your Current State Nodes to the New Pattern

State Node Storage Shape Sync Pattern Migration
TurnNumberNetworkedState _value: int Scalar → rpc() ✅ No change needed
TurnOrderNetworkedState _value: Array[int] Scalar → rpc() ✅ No change needed
ActivePlayerNetworkedState _value: int Scalar → rpc() ✅ No change needed
ActionPointsNetworkedState Needs: Dictionary[int, int] Per-player public → rpc(peer_id, val) ⚠️ Migrate to Variant A
HandNetworkedState (hands.gd) Needs: Dictionary[int, Array] Per-player private → rpc_id(owner) + count rpc() ⚠️ Implement as Variant B

VI. The 5 Inviolable Rules for Per-Player Networked State

  1. Dictionary[peer_id → value] is the standard. It is backed by official Godot docs, multiple community guides, and your own existing GameState design. Use it everywhere state varies per player.
  2. Use multiplayer.get_remote_sender_id() — never trust a peer_id argument in a request. A malicious client can pass any peer_id as an argument. The server always derives identity from the actual sender.
  3. Private state must never hit rpc(). Use rpc_id(target_peer_id, value). The only thing broadcast to all peers about a private hand is its count.
  4. Clients store only what they need. A client's local HandNetworkedState holds their own hand, period. They don't need a Dictionary client-side — just a local _value: Array.
  5. Keep the request/apply/sync seam. The seam is still valid for per-player state — request_spend(amount) identifies the spender from get_remote_sender_id(), _apply_set(peer_id, val) stores and syncs. No shape change needed on the seam itself.
