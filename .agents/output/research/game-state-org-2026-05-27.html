Organizing the Request → Validate → Apply → Sync Stack in Game Managers

The Core Question: You have a 4-method pipeline per piece of state: request_* → _validate_* → _apply_* → _rpc_sync_*. Three candidate organizations: (A) group the full stack per state vertically in one file, (B) one file per state variable, (C) one file per method type (all requests together, all applies together). What do successful devs do?

I. The Success/Failure Matrix

Organization Style Initial Feel At 5+ State Variables Adding a New State Variable Debugging a Bug in One Variable
A — Vertical grouping (full stack together, one file) Slightly more scrolling than horizontal One logical section per state; very readable Add one contiguous block; no other sections touched One section, everything visible — minimal context switching
B — File per state variable More files from day one Clean per-file clarity; easy to delete a whole state Create new file; wire to parent manager One file open, fully focused — strongest isolation
C — File per method type (all requests in one file, etc.) Seems organized Each file grows proportionally; files become long Touch 4+ files simultaneously — classic "shotgun surgery" Jump between 4 files to trace one pipeline

Drop-off Point for Style C: The moment you need to trace a bug in AP spend validation, you have your editor open to requests.gd, validators.gd, appliers.gd, and rpc.gd simultaneously. At this point most devs report regretting the structure. Martin Fowler calls this the Shotgun Surgery smell: "every time you make a change, you have to make many little changes to a lot of different classes." [Fowler, Refactoring Catalog]

II. The Cognitive Load Problem

The bottleneck in maintaining a networked game manager is cognitive locality — the ability to hold all relevant logic for one piece of state in working memory at once. All three options produce the same running code. They differ entirely in how much mental overhead you carry while reading or changing them.

Horizontal (Style C) — Highest Cognitive Load

Organizing by method type mirrors how relational databases are normalized — technically clean, practically expensive for humans. The vertical slice architecture literature explains the failure mode clearly: "The cohesion is low because you are creating many files in different layers. You would need to ensure that every representation of the model is updated or added accordingly, which poses a certain risk for missing one of the many folders." [AntonDevTips — Vertical Slice Architecture]

The CQRS community has lived this problem at scale. When every new command required touching a separate handler, validator, repository, and controller layer simultaneously, the pattern devolved into what the community called Anemic Domain Model — technically separated but logically incoherent. [Architecture Weekly — Vertical Slices & CQRS]

Vertical Grouping (Style A & B) — Cognitive Locality Preserved

Jimmy Bogard's foundational principle for vertical slice architecture: "Minimize coupling between slices, and maximize coupling in a slice." Your state variable IS the slice. Everything that touches AP — request, validate, apply, sync — belongs in one cohesive unit. [Architecture Weekly]

The feature-slice literature enumerates the cognitive benefits directly: "We drill down the folders by feature (and subfeatures), and the main components in charge reveal themselves… Feature slices maintain developer focus. Since all components for a feature live nearby, developers avoid the distraction of constant layer-switching and can sustain 'the flow' more effectively." [7 Reasons for Feature Slices — DEV Community]

III. The "Working" Stack — Three Validated Approaches

Method Theoretical Basis Practical Implementation
Style A: Vertical Groups Within One File (Recommended starting point) Vertical Slice Architecture. Cohesion per feature/state, not per layer. From Jimmy Bogard's VSA design: "minimize coupling between slices." [AntonDevTips] In game_manager.gd, add section comments like # === AP STATE === and place request_spend_ap → _validate_spend_ap → _apply_spend_ap → _rpc_sync_ap contiguously. Getters and helpers for that state live in the block too. A new engineer reads top-to-bottom and understands all of AP in one pass.
Style B: File Per State Variable (Correct when file grows too large) Single Responsibility Principle + composition. Aligns with CQRS Scalable Aggregates pattern where each aggregate owns its pipeline rather than accumulating all handlers centrally. [CQRS Scalable Aggregates] Extract ApState.gd, TurnState.gd, CurrencyState.gd as inner objects or class resources. GameManager becomes a thin coordinator that holds instances and delegates. The threshold to split: when the manager file exceeds ~400 lines, or when you need to unit-test one state variable in complete isolation from the others.
Style C: File Per Method Type (Not recommended) Horizontal layering. Matches how enterprise back-ends are sometimes organized (Controller / Service / Repository). Works in teams with strict layer ownership. [Greg Park — Horizontal vs Vertical Slices] Would mean requests.gd, validators.gd, appliers.gd, rpc_syncs.gd. Avoid for a solo or small team. The "shotgun surgery" cost hits every state change. Only justified when different programmers are contractually responsible for different layers (e.g., dedicated netcode engineer owns rpc_syncs.gd).

IV. The Game Networking Angle: Why "By State" Is the Right Default

In authoritative-server game networking, each state variable has a trust boundary: client requests are untrusted, server validation is the gate, server apply is authoritative, and sync propagates truth. This boundary is per state variable, not per method type. Your validation logic for AP spend has nothing to do with your validation logic for turn increment. They share no invariants. Grouping them together in a validators file creates false cohesion.

The Godot 4 authoritative server guide makes this explicit: "Never trust anything the client sends except input. Every any_peer RPC must verify the sender matches the action being performed." [StraySpark — Godot 4 Authoritative Server] The validation for AP is about AP's rules. Keeping that validator adjacent to AP's applier means a future maintainer can verify correctness without ever leaving the AP section.

The generalist programmer networking guide confirms the separation of concerns follows state boundaries: "Only validated server state affects gameplay fairness, while client prediction handles responsiveness." [Game Networking Fundamentals 2025] The natural unit of reasoning is the state variable, not the method type.

The litmus test: If someone asks "how does AP spending work?", can they find the answer by reading a single contiguous block of code? With Style A: yes. With Style C: they must mentally stitch together 4 files.

V. The 5 Inviolable Laws

Law 1: Organize by state variable, not by method type.

Every piece of state — AP, turn, active player, currency — owns its complete pipeline: request → validate → apply → sync. These four methods are more coupled to each other than to their counterparts in other state variables. File/section structure must reflect that coupling. [Architecture Weekly]

Law 2: Start vertical-in-one-file; split to files only when a state variable's complexity justifies it.

Early splits create premature abstraction. VSA literature warns: "Feature slices discourage premature abstraction… each slice can remain simple initially, evolving only to necessary complexity." [DEV Community] The trigger for splitting is file size or independent testability, not anticipation.

Law 3: Never split by method type unless teams own layers.

Horizontal (layer-per-file) only pays off when separate engineers own separate layers. For a solo or small-team game project, it forces shotgun surgery on every feature addition. "Changes that affect the same behavior are scattered across many files rather than isolated to one place." [AntonDevTips]

Law 4: Validators live next to appliers, not in a validators region.

A validator for AP spend is authoritative about AP business rules. It belongs adjacent to _apply_spend_ap, not near validators for turn management. The shared fact between them is the pipeline shape, not the domain. Keep domain cohesion above pipeline cohesion. [Game Networking Fundamentals 2025]

Law 5: The unit of change is the state variable, not the method type.

When requirements change (e.g., "AP can now overflow by 2 during a power-up phase"), you change one section: the AP block. When you add a new state variable (e.g., momentum), you add one new block. "The cohesion for a single use case is very high. This simplifies the development experience and makes it easy to find all the relevant components." [AntonDevTips]

VI. Concrete Recommendation for Your GameManager

Given the current file size (~423 lines) and number of state variables (AP, turn, active player, currency, player hand/deck/discard), the optimal structure is Style A — vertical groups within one file, organized like this:

game_manager.gd

# === LIFECYCLE ===
# _ready, _exit_tree

# === TURN STATE ===
# request_reset_turn, request_increment_turn
# (no validation needed yet — server-only, internal)
# _apply_set_turn, _apply_increment_turn
# _rpc_sync_turn

# === AP STATE ===
# request_spend_ap
# _validate_spend_ap   <-- NEW: extracted from inline check
# _apply_spend_ap, _apply_add_ap
# _rpc_sync_ap
# can_spend_ap          <-- stays here, it's AP's business rule

# === ACTIVE PLAYER STATE ===
# set_active_player
# _apply_active_player, _apply_switch_turn
# _rpc_sync_active_player

# === CURRENCY STATE ===
# add_currency
# _apply_add_currency
# _rpc_sync_player_currency

# === PLAYER COLLECTION STATE (hand / deck / discard) ===
# draw_cards
# _apply_draw_cards
# sync_player_hand_to_all_peers
# _rpc_sync_player_hand, _rpc_sync_player_deck, _rpc_sync_player_discard

# === PLAYER REGISTRY ===
# initialize_all_players, _initialize_player
# _rpc_register_player, _rpc_sync_player_number

# === FSM ===
# transition_to_phase

# === HELPERS ===
# _is_player_1, _get_next_player_id, _get_opponent_id, etc.

When to graduate to Style B (file per state): If any single state block — AP, for example — grows to require 80+ lines of logic (complex overflow rules, card effect modifiers, multi-step validation chains), extract it to src/levels/game/state/ap_state.gd with a matching class_name ApState. GameManager then holds an _ap: ApState instance and delegates to it. The pipeline shape stays identical; only the file boundary moves.
