Per-State-File Design, Public Getter / Private Setter, and the Caller-Guard Question

The Design Under Review: Move each piece of game state (ap, turn, active_player, etc.) to its own file. GameManager holds public instances: game_manager.ap. Reads via game_manager.ap.value. Mutations via game_manager.ap.request_spend(amount). Optional: a setter guard on value that enforces the write came from the right context. Question: where is the line between tight and over-engineered?

I. The Core Pattern Is Already Named: DDD Aggregate Root

What you're describing has a formal name in Domain-Driven Design: the Aggregate Root. The pattern's defining rule is: "All modifications to the aggregate must go through its root entity, allowing the aggregate to maintain invariants and uphold business rules." [MomentsLog — Aggregate Roots]

The implementation mechanics are exactly what you described: "Aggregates and Entities usually have public getters but private setters." This creates three layers: [BuildSimple — DDD Aggregates in Practice]

  1. Private internal state — _value: int — writable only from within the class
  2. Public read-only access — value getter — exposes current state without mutation capability
  3. Public mutation methods — request_spend(amount) — the only legitimate entry point for changes

This is battle-tested across game and enterprise architectures. The DDD literature confirms: "External interactions must occur through clearly defined operations provided by the aggregate root." [MomentsLog]

II. The Critical GDScript Constraint: @rpc Requires Node

Hard technical limit: In Godot 4, @rpc is implemented via MultiplayerAPI.rpc(), which requires the object to be a Node in the active scene tree. It will fail on RefCounted and plain Object classes. [The Shaggy Dev — RefCounted in Godot 4] [Godot Engine — RPC Syntax Report]

This forks the design into two valid approaches. The choice determines how much pipeline each state file can own:

Approach Base Class Can own @rpc? Scene tree cost Full self-containment?
A — Node-based state extends Node, added as named child of GameManager ✅ Yes — full pipeline in one file One extra node per state variable ✅ Complete: request → validate → apply → rpc_sync all live in the state file
B — RefCounted state extends RefCounted, held as a member variable ❌ No — @rpc stays in GameManager None 🟡 Partial: request + validate + apply in state file; rpc_sync stays in GameManager and calls back into state via sync_from_authority(v)

Recommendation: Approach A (Node-based). The node overhead is trivial. In exchange you get complete self-containment — the entire AP pipeline is readable in ap_state.gd without opening GameManager at all. This fully realizes the intent of moving to per-state files.

III. What GDScript 4 Properties Can Actually Enforce

In GDScript 4, properties use get: / set(v): blocks inline on the variable declaration. Three things are confirmed true:

1. A getter-only property is read-only to outside callers

var value: int:
    get:
        return _value
# No "set:" block = any external assignment is a compile-time error in typed mode
# Internal code can still write _value directly

2. A setter can contain any logic including multiplayer.is_server()

var _value: int:
    set(v):
        assert(multiplayer.is_server(), "Only server may write AP")
        _value = v

Setters are executed on every assignment to the variable, including from within the same class. [Godot Engine — Multiplayer 4.0 Report]

3. GDScript has no private access modifiers

The _ prefix on _value is convention only — not enforced by the compiler. Any external code can write ap_state._value = 999 and it will work at runtime. The protection is social, not structural.

Bottom line on setter enforcement: GDScript gives you good-faith encapsulation, not hard encapsulation. The var value: int: get: return _value pattern plus assert(multiplayer.is_server()) in _apply_* is the practical ceiling of enforcement this language can offer.

IV. The Caller-Guard Question — Exactly Where the Line Is

The user's instinct: "build an actual setter that verifies we came from an rpc." This is worth unpacking precisely, because it splits into several sub-ideas with different viability profiles.

Guard Idea Possible in GDScript? Verdict
Getter-only public property (value has no set:) ✅ Yes — standard syntax DO IT This is free and catches typos at compile time
assert(multiplayer.is_server()) inside _apply_* ✅ Yes — runtime check DO IT Crashes loudly in debug if a client somehow reaches apply. Zero overhead in release builds
A setter on _value that checks multiplayer.is_server() ✅ Yes — technically valid SKIP Adds noise. You already guard in _apply_*. Double-guarding via the setter makes stack traces harder to read and fires on every internal write, including test setup
_is_applying: bool flag — set before _apply_*, checked in setter ✅ Yes — possible to implement OVER-ENGINEERING Mutable state used purely for structural enforcement. Introduces a new failure mode (flag not reset on error path). DDD warns: this creates "impressive amounts of artificial complexity." [BuildSimple]
Call stack inspection — verify the setter was called from _rpc_sync_* ❌ No — GDScript has no call stack introspection API IMPOSSIBLE GDScript has no equivalent of Python's inspect.stack() or Java's getStackTrace()

V. The Reference Implementation — ApState.gd

This is what the pattern looks like when applied correctly to AP, using the Node-based approach. The entire pipeline lives in one file. The guard is clean. Nothing over-engineered.

class_name ApState
extends Node
## Owns the full request → validate → apply → sync pipeline for the AP tracker.
## future @rpc("any_peer") on request_spend, @rpc("authority") on _rpc_sync_value

## Read-only externally. No set: block = external assignment is a compiler error.
var value: int:
    get:
        return _value

var max_value: int = 10
var _value: int = 0


# Request — future @rpc("any_peer", "reliable")
func request_spend(amount: int) -> void:
    if not multiplayer.is_server():
        return  # future: rpc_id(1, "request_spend", amount)
    if not _validate_spend(amount):
        return
    _apply_spend(amount)


# Validate
func _validate_spend(amount: int) -> bool:
    if amount <= 0:
        push_error("AP spend amount must be positive: %d" % amount)
        return false
    if amount > _value + max_value:
        push_error("Cannot spend more AP than available: %d" % amount)
        return false
    return true


# Apply — the server guard lives here, not in the setter
func _apply_spend(amount: int) -> void:
    assert(multiplayer.is_server(), "ApState._apply_spend called on non-server peer")
    _value = clamp(_value - amount, -max_value, max_value)
    _rpc_sync_value.rpc(_value)


# Sync — future @rpc("authority", "reliable")
@rpc("authority", "reliable")
func _rpc_sync_value(v: int) -> void:
    _value = v
    SignalBus.ap_synced.emit(_value)

In GameManager:

@onready var ap: ApState = $ApState  # Added as named child node in the scene

Callers anywhere:

# Read
var current_ap: int = game_manager.ap.value

# Mutate (routes through the full pipeline)
game_manager.ap.request_spend(2)

VI. The 5 Laws

Law 1: State classes extend Node so they can own their @rpc methods.

RefCounted cannot use @rpc. Node-based state files achieve true self-containment of the full pipeline. The overhead is one extra scene tree node per state variable — trivial. [Shaggy Dev]

Law 2: Public getter, no setter on value.

Define var value: int: get: return _value with no set: block. External callers get a compiler error if they try to write. This is the cleanest, lowest-cost setter guard available in GDScript. [MomentsLog — Aggregate Roots]

Law 3: The server guard lives in _apply_*, not in a setter.

Use assert(multiplayer.is_server()) at the top of every _apply_* method. Asserts crash loudly in debug builds and cost nothing in release. Putting the guard in the setter instead doubles up without adding real protection, and fires on legitimate internal writes like test setup.

Law 4: Stop before the _is_applying flag.

The idea of a flag that tells the setter "a valid apply method called me" is over-engineering. It adds mutable state for a structural purpose, introduces an entire class of reset-on-error bugs, and DDD practitioners warn that this kind of enforcement creates "impressive amounts of artificial complexity" when domain complexity doesn't justify it. [BuildSimple]

Law 5: Call stack verification is impossible in GDScript — don't design around it.

GDScript has no runtime call stack introspection API. There is no equivalent of Python's inspect.stack() or Java's Thread.currentThread().getStackTrace(). Designs that assume this capability will either require C++ engine extensions or won't be buildable at all. The protection ceiling in pure GDScript is: no public setter + server assert in apply methods.
