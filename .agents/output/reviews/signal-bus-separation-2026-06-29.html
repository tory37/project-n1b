SignalBus Separation & NotificationManager Echo Loop

Summary

This change splits the monolithic SignalBus into client- and server-scoped buses (ClientSignalBus, ServerSignalBus) and pulls game-manager responsibilities out into SubGameManager subclasses (CardManager, NotificationManager). The FSM phases and UI now talk to the correctly-scoped bus instead of going through fat methods on GameManager. The direction is solid: intent is clearer, the server/client boundary is explicit, and game_manager.gd shrank from a god object to a thin coordinator.

There is one confirmed bug and it is exactly the symptom you described: the NotificationManager creates a self-perpetuating RPC echo loop because its signal handler is connected on every peer, not just the server.

Critical Findings

CRITICAL NotificationManager re-broadcasts its own RPC → infinite echo loop

Root cause — confirmed by code inspection with a complete causal chain.

Unlike CardManager (which guards _ready/_exit_tree with if not multiplayer.is_server(): return), NotificationManager._ready() connects _on_notification_fired unconditionally — so the handler is live on the server and on every client. The handler's only job is to re-broadcast via emit_game_notification.rpc(message), and the RPC body emits the very same SignalBus.notification_fired signal that the handler listens to. That closes a loop:

  1. Something emits notification_fired("X") (e.g. a server FSM action, or message_manager on a client).
  2. The local NotificationManager._on_notification_fired fires → emit_game_notification.rpc("X") broadcasts to all other peers.
  3. On a receiving client, emit_game_notification sees is_server() == false and calls SignalBus.notification_fired.emit("X").
  4. That local emit re-triggers that client's own NotificationManager._on_notification_fired → it RPCs again → other clients emit again → … ping-pong forever with 2+ clients.

Each re-emit also re-fires notification_ui._on_notification_fired, which calls show() + restarts the 3s Timer on every bounce. Rapid restarts of the show/fade is precisely the "blurs as if multiple notifications are fighting" symptom — the panel is being re-shown dozens of times per frame. (The server peer breaks its own leg of the loop via the is_server() → return guard, but client→client legs have no such guard, so the loop never terminates.)

Secondary issue on the same handler: message_manager.gd emits notification_fired("Your Turn." / "Opponent's Turn.") locally on each client (it keys off multiplayer.get_unique_id()). Because the client's NotificationManager is connected, that purely-local message also gets RPC-broadcast to everyone — redundant, since every client already computes its own turn message independently.

Fix (mirror the CardManager pattern — make NotificationManager a server-only broadcaster): 

func _ready() -> void:
	if not multiplayer.is_server():
		return
	SignalBus.notification_fired.connect(_on_notification_fired)

func _exit_tree() -> void:
	if not multiplayer.is_server():
		return
	SignalBus.notification_fired.disconnect(_on_notification_fired)

After this: server-originated notifications broadcast once → each client's emit_game_notification emits locally → only notification_ui is listening on the client → shown once, no re-broadcast, no loop. Client-local messages (message_manager) stay local and stop double-broadcasting. The _exit_tree guard is required too — without it, clients would call disconnect on a signal they never connected, which errors in Godot.

What would disprove this: if NotificationManager._ready() were actually only ever instantiated on the server, the loop couldn't form — but the node lives under $SubManagers in game.tscn, which is part of the scene on every peer, and only setup() (not _ready's connect) is gated to the server in game_manager.gd. So the connect runs everywhere. To fully confirm at runtime: log peer id inside _on_notification_fired and play a card with 2 clients — you'll see the same message bounce between client ids.

SUGGESTION CardManager._exit_tree disconnects a signal it may never have connected

_on_card_play_disabled is connected lazily inside _on_card_play_enabled (line 27) and disconnected inside the handler (line 32). But _exit_tree (line 18) unconditionally disconnects card_play_disabled too. If a player never had card-play enabled this round (or it already fired and self-disconnected), the _exit_tree disconnect targets a connection that isn't there → Godot error on teardown. Consider is_connected() guards, or only disconnect what _ready connected. Same hazard class as the NotificationManager fix.

Positive Notes

ServerSignalBus carries peer-targeted authority signals (card_play_enabled(peer_id), card_draw_requested(player_id, count), card_play_validated(card)); ClientSignalBus carries presentation/request signals. The FSM phases now emit intent (ServerSignalBus.card_draw_requested.emit(...)) instead of reaching into GameManager.draw_cards() — that decouples the state machine from the manager's implementation and is exactly the separation you're going for. UI files consistently moved to ClientSignalBus. Clean, legible diff.

Test Coverage

No automated tests accompany the change (consistent with the project's current setup). The echo loop is the kind of defect a 2-client integration smoke test would catch immediately. At minimum, manually verify with two connected clients after the fix.

  • Play a card with 2 clients → notification appears once per client, no flicker/blur.
  • Turn change → "Your Turn." / "Opponent's Turn." shows once on the correct client, not broadcast.
  • Leave/teardown a game mid-round → no Godot disconnect errors in the console (covers the CardManager _exit_tree note).
  • Missing: any automated regression guard for signal-loop / RPC-echo behavior.
