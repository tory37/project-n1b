REVIEW 2026-05-27

Godot Best Practices Review — TurnNumberState

Scope Network state template (first implementation)
Branch game_state_overlay
Intent Copy-paste template for all encapsulated network state items

Summary

The structural skeleton is sound — the request/apply/sync layer separation is the right model, the validation hook is a good idea, and the lifecycle connect/disconnect pairing is correct in concept. However, there are 5 critical bugs that will cause immediate runtime failures, and 4 template-quality issues that — if baked into a copy-paste template — will silently introduce the same problems into every future state class. Fix all of these before using this as a template.

Critical Findings

CRITICAL Category A — GDScript / Category F turn_number.gd:8 — Infinite recursion in value getter

The computed property getter reads return value, which calls the getter again recursively. This will stack-overflow the moment anything reads value.

# BUG
var value: int:
    get:
        return value   # ← calls itself

# FIX
var value: int:
    get:
        return _value  # ← reads the backing field
CRITICAL Category M — RPC / Category C — Signals turn_number.gd:38 — Wrong signal name; handler is never called

signal_bus.gd defines the signal as increment_turn_requested, but this file connects to "turn_increment_requested". The words are transposed. The connection silently fails — no error, no handler, dead code.

# signal_bus.gd:11
signal increment_turn_requested(turn: int)

# BUG — wrong name, never fires
SignalBus.connect("turn_increment_requested", _on_turn_increment_requested)

# FIX — use property-access style (GDScript 2.0, compile-time checked)
SignalBus.increment_turn_requested.connect(_on_turn_increment_requested)

Same fix applies to the _exit_tree disconnect on line 42.

CRITICAL Category C — Signals turn_number.gd:75 — Wrong failure signal name; emits to void

signal_bus.gd defines increment_turn_requested_failed, but this file emits "turn_increment_failed". Emitting a signal that doesn't exist on the bus throws a runtime error.

# signal_bus.gd:12
signal increment_turn_requested_failed()

# BUG — signal does not exist on SignalBus
SignalBus.emit_signal("turn_increment_failed")

# FIX — use property-access emit (GDScript 2.0)
SignalBus.increment_turn_requested_failed.emit()

Also applies to line 70: SignalBus.emit_signal("turn_synced", new_value) should be SignalBus.turn_synced.emit(new_value). turn_synced at least matches a real signal, but the string API is deprecated.

CRITICAL Category M — RPC turn_number.gd:46 — Argument passed to zero-parameter RPC

The signal handler receives turn: int from increment_turn_requested and forwards it as request_increment.rpc(turn). But request_increment() declares no parameters — it calculates _value + 1 internally. Passing a mismatched argument count to an RPC call is undefined behaviour and will produce a Godot runtime warning or error.

# BUG — extra arg sent to zero-param RPC
func _on_turn_increment_requested(turn: int) -> void:
    request_increment.rpc(turn)

# FIX — drop the unused argument
func _on_turn_increment_requested(_turn: int) -> void:
    request_increment.rpc_id(1)

Note the switch to rpc_id(1) — see warning below about .rpc() vs .rpc_id(1) on the client.

CRITICAL Category B — Networking Seam / Category M — RPC turn_number.gd:13 — request_set missing @rpc; breaks architecture seam

The architecture mandate states: "request_* — public entrypoint, callable by any peer. Will become @rpc("any_peer") when networking is added." request_set has no @rpc decorator, which means the seam doesn't exist — adding networking later requires rewriting this function, not decorating it. As a template, this omission will be copy-pasted into every future state class.

# BUG — no seam for future @rpc
func request_set(new_value: int) -> void:
    ...

# FIX — seam present; adding networking = adding the decorator
@rpc("any_peer", "reliable")
func request_set(new_value: int) -> void:
    if not multiplayer.is_server():
        return
    _apply_set_value(new_value)

Warnings

WARNING Category B — Networking Seam turn_number.gd:59 — _apply_set_value doesn't set _value before broadcasting

The established GameManager pattern is: set state locally first, then broadcast. _apply_set_turn in game_manager.gd:218 does _game_state.turn = turn before calling _rpc_sync_turn.rpc(turn). _apply_set_value here skips the local set entirely, relying on the RPC running locally via Godot's authority-local-call behaviour.

This technically works today but is fragile and inconsistent. If call_local semantics change, or a future refactor skips the RPC under some condition, _value on the server will be stale. Match the established pattern:

# FIX — explicit local set, then broadcast
func _apply_set_value(new_value: int) -> void:
    _value = new_value
    _rpc_sync_value.rpc(new_value)
WARNING Category M — RPC turn_number.gd:46 — Client should use rpc_id(1), not rpc()

On the client, calling request_increment.rpc() broadcasts to all peers, including other clients. The correct pattern for a client forwarding a request to the server is request_increment.rpc_id(1) — targeting peer ID 1 (the server) specifically. This mirrors what game_manager.gd:74 does: request_spend_ap.rpc_id(1, amount).

# BUG — broadcasts to all peers
request_increment.rpc()

# FIX — sends only to server
request_increment.rpc_id(1)
WARNING Category B — Networking Seam turn_number.gd:66,73 — RPC sync naming diverges from codebase convention

GameManager uses the prefix _rpc_sync_* for all authority-broadcast RPCs (e.g. _rpc_sync_turn, _rpc_sync_ap). This file uses _sync_*. As a template, this inconsistency will propagate. Rename to _rpc_sync_value and _rpc_sync_increment_failure.

WARNING Category A — GDScript Standards turn_number.gd:38,42,70,75 — String-based signal API (GDScript 1.x style)

All four signal calls use the old string-argument form (SignalBus.connect("name", ...), SignalBus.emit_signal("name", ...)). GDScript 2.0 uses the property-access form which is compile-time type-checked and will catch mismatched names before runtime. Two of the three bugs above (findings #2 and #3) would have been caught by the compiler if property-access style had been used.

# GDScript 1.x — not compile-time checked
SignalBus.connect("turn_synced", _on_foo)
SignalBus.emit_signal("turn_synced", value)

# GDScript 2.0 — compile-time checked, preferred
SignalBus.turn_synced.connect(_on_foo)
SignalBus.turn_synced.emit(value)
WARNING Category L — File Layout turn_number.gd — Lifecycle methods appear after public methods

Godot 4's official declaration order puts built-in lifecycle methods (_ready, _exit_tree) before signal callbacks and before public methods. Currently _ready and _exit_tree appear after request_set and request_increment. The correct order for this file:

  1. class_name / extends
  2. Variables (_value, value)
  3. Lifecycle: _ready, _exit_tree
  4. Signal callbacks: _on_turn_increment_requested
  5. Public methods: request_set, request_increment
  6. Private methods: _validate_set_value, _apply_set_value, _rpc_sync_*

Suggestions

SUGGESTION Category A — GDScript Standards turn_number.gd:51 — Remove underscore from _new_value parameter

The leading underscore on _new_value signals "intentionally unused." For a validation stub that is explicitly meant to be filled in, the underscore is misleading. Use new_value.

SUGGESTION Category A — GDScript Standards turn_number.gd:59 — Redundant server guard in _apply_set_value

Both callers (request_set and request_increment) already guard with if not multiplayer.is_server(): return before calling _apply_set_value. The guard inside _apply_set_value is dead code under normal usage and adds noise. GameManager's _apply_* functions do not repeat the guard — the server check lives in request_*. Remove it from _apply_set_value and add a comment documenting the precondition if desired.

SUGGESTION Category M — RPC turn_number.gd:23 — request_increment silently returns on client; should log

request_set calls Loggit.p(..., "Error") when the server check fails. request_increment silently returns. For a template, be consistent — either both log or neither does. Logging is preferable for debuggability.

SUGGESTION Category F — Node vs. Base Classes turn_number.gd:2 — Consider whether extends Node is correct

This class holds a single primitive value and a set of network methods. It has no visual presence and no physics. extends Node is correct only if it needs to live in the scene tree to call multiplayer.is_server() and receive RPC messages — which it does. The extends Node is justified, but it's worth explicitly noting this in a comment as a template convention so future authors don't wonder if it should be RefCounted.

Positive Notes

POSITIVE Layer separation is correct

The three-layer split — request (entrypoint) → apply (mutation) → sync (broadcast) — is exactly right. _apply_set_value correctly has no @rpc annotation. _sync_value correctly uses @rpc("authority", "reliable"). The skeleton is solid.

POSITIVE Validation hook is a well-designed template slot

_validate_set_value as a dedicated private method is a clean separation. Future states can override validation logic without touching the request/apply flow. The stub pattern (returns true by default) is a good no-op default.

POSITIVE _ready / _exit_tree connect/disconnect pairing

Disconnecting in _exit_tree is correct practice — it prevents dangling signal connections if the node is removed from the tree mid-game. Many implementations forget this.

POSITIVE Failure path is modelled

Having a _rpc_sync_increment_failure RPC that notifies clients of a rejected request is the right design. Most simple state templates skip this entirely. Keeping it in the template ensures every future state class has a failure signal slot from day one.
