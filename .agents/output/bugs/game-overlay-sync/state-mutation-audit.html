TECHDEBT 2026-05-27

State Mutation Audit — Direct Writes Outside _apply_*

Context

The request_* / _apply_* pattern is the project's networking seam: every state change is supposed to go through an _apply_* method that both writes local state and fires the corresponding RPC to keep all peers in sync. Mutations that bypass this seam produce divergent client state with no error — silent, hard-to-trace bugs.

The game overlay sync bugs were caused by exactly this pattern. This audit catalogs every other place where it happens.

Scope: src/levels/game/game_manager.gd and the game phase files.

Findings

CRITICAL Finding 1 — current_phase written directly, never synced

transition_to_phase writes _game_state.current_phase directly and has no RPC to push the value to clients. There is no _apply_current_phase and no _rpc_sync_current_phase. 

// game_manager.gd:243
func transition_to_phase(phase: GameState.Phase) -> void:
    _game_state.current_phase = phase           // ← raw write, no RPC
    _game_fsm.change_state(...)

Clients can never know what phase the game is in. Any client-side logic that needs to gate on phase (e.g. "can I play a card right now?") reads stale data.

game_manager.gd:243
CRITICAL Finding 2 — _initialize_player makes three unsynced writes

_initialize_player sets up all initial per-player state on the server but routes none of it through _apply_* methods. Three fields are affected:

Field Line Impact
player_id_turn_order.append(peer_id) 454 Never synced — root cause of the overlay player-number bug (Bug 1 in fix-plan)
player_states[peer_id].deck = new_deck 451 Deck set server-side only; client deck is empty until game_phase_start manually calls sync_player_deck_to_all_peers
player_states[peer_id].player_number = ... 452 Written locally; the external call to sync_player_number that was supposed to push it is broken (Bug 2 in fix-plan)

All three should be handled by a single _apply_initialize_player(peer_id) that also fires one broadcast RPC to register the peer with its full initial state on all clients.

game_manager.gd:446–454
WARNING Finding 3 — _apply_teardown_player erases state without an RPC

The method follows the naming convention but is missing its RPC half. It erases the player from player_states on the server only — clients keep a stale entry for the departed player. 

// game_manager.gd:313
func _apply_teardown_player(peer_id: int) -> void:
    _game_state.player_states.erase(peer_id)   // ← no _rpc_teardown_player.rpc(...)

Lower priority since teardown only fires on disconnect, but it will cause ghost-player state on reconnect or lobby-return flows.

game_manager.gd:313–314
WARNING Finding 4 — _apply_draw_cards syncs hand but not deck

deck_to_hand() modifies both the deck and the hand. _apply_draw_cards calls sync_player_hand_to_all_peers but has no corresponding deck sync. The deck count on all clients goes stale after every draw. 

// game_manager.gd:317
func _apply_draw_cards(player_id: int, count: int) -> void:
    _game_state.player_states[player_id].deck_to_hand(count)  // mutates deck AND hand
    sync_player_hand_to_all_peers(player_id)                  // ← only hand synced
    // missing: sync_player_deck_to_all_peers(player_id)

This is currently masked in game_phase_start.gd which manually calls sync_player_deck_to_all_peers after draw_cards. But game_phase_draw_card.gd only calls draw_cards — so the client deck count is never updated during the draw phase.

game_manager.gd:317–332, game_phase_draw_card.gd:13

Summary

# Location Field(s) Severity Currently Broken?
1 transition_to_phase current_phase Critical Yes — clients never know the current phase
2a _initialize_player player_id_turn_order Critical Yes — overlay bug (fix-plan Bug 1)
2b _initialize_player player_states[x].deck Critical Partially — masked by explicit call in start phase
2c _initialize_player player_states[x].player_number Critical Yes — broken by fix-plan Bug 2
3 _apply_teardown_player player_states (erase) Warning Not yet — disconnect flow not implemented
4 _apply_draw_cards player_states[x].deck Warning Yes — deck count stale after draw_card phase

Recommended Rule

Every write to _game_state must live inside an _apply_* method. Every _apply_* method must call the corresponding _rpc_* (or a sync_* wrapper) before returning. No code outside GameManager may touch _game_state at all — phase states go through the public API only.

Enforcing this eliminates the entire class of "forgot to sync" bugs at the architectural level.
