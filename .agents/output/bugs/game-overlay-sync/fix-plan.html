BUG 2026-05-27

Game Overlay Sync — Client Receives No Hand/Deck/Discard; All Updates Attributed to Player 2

Observed Symptoms

On CLIENT peer 1067925686, only four signals are received during game start:

  • active_player_synced — Player 1
  • ap_synced — AP: 0
  • currency_synced — Player 2 currency: 2 (appears twice)

Missing: player_number_synced, turn_synced, player_hand_synced, player_deck_synced, player_discard_synced.

The P1 currency label is never set. The P2 currency label is updated twice (once for each player's currency, both mislabeled as P2).

Root Cause Analysis

CONFIRMED Bug 1 — player_id_turn_order Never Synced to Clients

_initialize_player(peer_id) in game_manager.gd appends peer_id to _game_state.player_id_turn_order on the server only. The _rpc_register_player broadcast only creates player_states[peer_id] — it never touches player_id_turn_order.

On the client, player_id_turn_order is always []. Therefore, _is_player_1(any_id) always returns false (empty array → find() = −1), and _get_player_number() always returns 2.

Impact: Every signal that carries player_number (currency, hand, deck, discard) routes to the P2 UI labels. P1 labels are never updated. Player 1's currency update hits the P2 currency label — which is why the log shows "Player 2 currency: 2" twice (once per player).

game_manager.gd — _rpc_register_player (line 382) and _initialize_player (line 446)
CONFIRMED Bug 2 — sync_player_number Passes Wrong Argument Count to RPC

sync_player_number(player_id) calls: 

_rpc_sync_player_number.rpc_id(player_id, player_states[player_id].player_number)

rpc_id(target_peer, arg1) sends ONE argument (player_number value) to the target peer. But the receiving function expects TWO arguments: (player_id: int, player_number: int). Godot 4 will fail to decode this RPC call on the receiver, so _rpc_sync_player_number never executes on the client. player_number_synced is never emitted.

Additionally, sync_player_number(1) targets peer 1 (the server itself). The server does not have a GameOverlayUI, so even if it worked, the client never learns its own player number.

game_manager.gd — sync_player_number (line 143) and _rpc_sync_player_number (line 388)
CONFIRMED — code evidence Bug 3 — GameCardCollection and GameCard Are Not RPC-Serializable

Both classes extend RefCounted with no @export annotations on any property:

  • GameCardCollection: var cards: Array[GameCard] = [] — not exported
  • GameCard: var revealed: bool, var data: CardData — not exported

Godot 4's encode_variant (used by RPC) cannot serialize custom RefCounted objects across the network. The server-side calls to _rpc_sync_player_hand.rpc_id(...), _rpc_sync_player_deck.rpc_id(...), and _rpc_sync_player_discard.rpc_id(...) all fail silently at the encoding step — the client never receives these RPCs at all.

This explains the complete absence of any hand/deck/discard log entries on the client.

game_card_collection.gd (line 3–4), game_card.gd (line 4–5)
Not Yet Confirmed: Whether Bug 3 fails silently at the sending side (encoder error) or at the receiving side (decoder error). Requires a Godot console check during gameplay. Either way, the fix is the same.

Fix Plan

1 Sync player_id_turn_order to Clients
  1. In _rpc_register_player, append peer_id to _game_state.player_id_turn_order alongside the player_states registration. Reliable RPCs are ordered, so player 1 is registered before player 2 on all peers — the order is consistent. 
    @rpc("authority", "reliable")
func _rpc_register_player(peer_id: int) -> void:
    if not _game_state.player_states.has(peer_id):
        _game_state.player_states[peer_id] = PlayerState.new()
        _game_state.player_id_turn_order.append(peer_id)  # ADD
Verify: After fix, _get_player_number(1) returns 1 and _get_player_number(1067925686) returns 2 on the client. Currency logs should show "Player 1 currency: 2" and "Player 2 currency: 2" separately.
2 Fix sync_player_number Argument Count
  1. Change sync_player_number to pass player_id as the first RPC argument, matching the function signature: 
    func sync_player_number(player_id: int) -> void:
    ...
    _rpc_sync_player_number.rpc_id(
        player_id,
        player_id,  # ADD — first function arg
        _game_state.player_states[player_id].player_number
    )
Verify: Client log shows "Received player number synced signal. Player number is now: 2".
3 Make GameCardCollection RPC-Serializable
  1. Add @export to all mutable properties in GameCard: 
    class_name GameCard
extends RefCounted

@export var revealed: bool = false
@export var data: CardData = null
  2. Add @export to the cards array in GameCardCollection: 
    class_name GameCardCollection
extends RefCounted

@export var cards: Array[GameCard] = []
  3. If Godot still cannot serialize the custom types after adding @export, the fallback is to add to_dict() -> Dictionary / from_dict(d: Dictionary) methods and change the 3 RPC functions to pass Dictionary instead of GameCardCollection. This will be determined after testing Phase 3.
Verify: Client log shows hand/deck/discard synced signals with correct card counts.
Side-effect risk: Adding @export to GameCard.data and GameCard.revealed may affect the Godot Inspector for any scenes that hold GameCard instances directly. Review after applying.
Redundant hand sync: game_phase_start.gd calls sync_player_hand_to_all_peers(player_id) explicitly after draw_cards(player_id, 5), which already calls it internally. This results in the hand being synced twice per player. Not a bug, but worth cleaning up as a follow-up.
